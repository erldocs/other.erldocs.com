iris_topic

A behavior module for implementing an Iris topic subscription handler.

A behavior module for implementing an Iris topic subscription handler. The behavior follows similar design principles to the OTP gen_server.

It is assumed that all handler specific parts are located in a callback module, which implements and exports a set of pre-defined functions. The relationship between the Iris API and the callback functions is as follows:

       iris_client module           callback module
       --------------               -------------------------
       iris_client:subscribe   ---> Module:init/1
       iris_client:publish     ---> Module:handle_event/2
       iris_client:unsubscribe ---> Module:terminate/2

If a callback function fails or returns a bad value, the iris_topic and parent iris_client or iris_server will terminate.

A slight difference to the gen_server behavior is that the event trigger methods have not been duplicated in the iris_topic module too, rather they use the methods defined in the iris_client module, as depicted in the above table.

Callback methods

Since edoc cannot generate documentation for callback methods, they have been included here for reference:

init/1 

       init(Args) -> {ok, State} | {stop, Reason}
           Args   = term()
           State  = term()
           Reason = term()

Whenever an iris_topic is started using iris_client:subscribe/4, this function is called in the new process to initialize the handler state.

Args is the Args argument provided to subscribe.

If the initialization succeeds, the function should return {ok, State}.

Otherwise, the return value should be {stop, Reason}

handle_event/2 

       handle_event(Event, State) -> {noreply, NewState} | {stop, Reason, NewState}
           Event    = binary()
           State    = term()
           NewState = term()
           Reason   = term()

Whenever an iris_topic receives an event sent using iris_client:publish/3, this function is called to handle it.

Event is the Event argument provided to publish.

State is the internal state of the iris_topic.

If the function returns {noreply, NewState}, the iris_topic will continue executing with NewState.

If the return value is {stop, Reason, NewState}, the iris_topic will call Module:terminate/2 and terminate.

terminate/2 

       terminate(Reason, State)
           Reason = term()
           State  = term()

This method is called when an iris_topic is about to terminate. It is the opposite of Module:init/1 and should do any necessary cleaning up. The return value is ignored, after which the handler process stops.

Reason is the term denoting the stop reason. If the iris_topic is terminating due to a method returning {stop, ...}, Reason will have the value specified in that tuple.

State is the internal state of the iris_topic.
