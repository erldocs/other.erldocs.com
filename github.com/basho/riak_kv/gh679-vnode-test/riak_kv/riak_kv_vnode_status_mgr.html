get_vnodeid_and_counter(Pid::pid(), LeaseSize::non_neg_integer()) -> {ok, {VnodeId::binary(), Counter::non_neg_integer(), LeaseSize::non_neg_integer()}}

You can't ever have a LeaseSize greater than the maximum 32 bit unsigned integer, since that would involve breaking the invariant of an vnodeid+cntr pair being used more than once to start a key epoch, the counter is encoded in a 32 bit binary, and going over 4billion+etc would wrap around and re-use integers.

lease_counter(Pid::pid(), LeaseSize::non_neg_integer()) -> ok

Asynchronously lease increments for a counter. Pid is the server pid, and LeaseSize is the number of increments to lease. A LeaseSize of 10,000 means that a vnode can handle 10,000 new key epochs before asking for a new counter. The trade-off here is between the frequency of flushing and the speed with which a frequently crashing vnode burns through the 32bit integer space, thus requiring a new vnodeid. The calling vnode should handle the response message of {counter_lease, {From :: pid(), VnodeId :: binary(), NewLease :: non_neg_integer()}}
