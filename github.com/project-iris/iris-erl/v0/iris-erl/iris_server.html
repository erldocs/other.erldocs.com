iris_server

A behavior module for implementing an Iris server event handler.

A behavior module for implementing an Iris server event handler. The behavior follows the exact same design principles as the OTP gen_server.

It is assumed that all handler specific parts are located in a callback module, which implements and exports a set of pre-defined functions. The relationship between the Iris API and the callback functions is as follows:

       iris_server module          callback module
       ----------------------      --------------------
       iris_server:start
       iris_server:start_link ---> Module:init/1
       iris_server:stop       ---> Module:terminate/2
       ---                    ---> Module:handle_drop/2
 
       iris module                 callback module
       --------------              -------------------------
       iris:broadcast         ---> Module:handle_broadcast/3
       iris:request           ---> Module:handle_request/4
       iris:publish           ---> Module:handle_publish/4
       iris:tunnel            ---> Module:handle_tunnel/3

If a callback function fails or returns a bad value, the iris_server will terminate. Unless otherwise stated, all functions in this module fail if the specified gen_server does not exist or if bad arguments are given.

A slight difference to the gen_server behavior is that the event trigger methods have not been duplicated in the iris_server module too, rather they use the methods defined in the iris module, as depicted in the above table.

Callback methods

Since edoc cannot generate documentation for callback methods, they have been included here for reference:

init/1 

       init(Args) -> {ok, State} | {stop, Reason}
           Args   = term()
           State  = term()
           Reason = term()

Whenever an iris_server is started using iris_server:start/4 or iris_server:start_link/4, this function is called in the new process to initialize the handler state.

Args is the Args argument provided to start/start_link.

If the initialization succeeds, the function should return {ok, State}.

Otherwise, the return value should be {stop, Reason}

handle_broadcast/3 

       handle_broadcast(Message, State, Link) -> {noreply, NewState} | {stop, Reason, NewState}
           Message  = binary()
           State    = term()
           Link     = iris:connection()
           NewState = term()
           Reason   = term()

Whenever an iris_server receives a message sent using iris:broadcast/3, this function is called to handle it.

Message is the Message argument provided to broadcast.

State is the internal state of the iris_server.

Link is the relay connection to the Iris network if the handler needs additional communication.

If the function returns {noreply, NewState}, the iris_server will continue executing with NewState.

If the return value is {stop, Reason, NewState}, the iris_server will call Module:terminate/2 and terminate.

handle_request/4 

       handle_request(Request, From, State, Link) ->
               {reply, Reply, NewState} | {noreply, NewState} |
               {stop, Reason, Reply, NewState} | {stop, Reason, NewState} |
           Request  = binary()
           From     = iris:sender()
           State    = term()
           Link     = iris:connection()
           Reply    = binary()
           NewState = term()
           Reason   = term()

Whenever an iris_server receives a request sent using iris:request/4, this function is called to handle it.

Request is the Request argument provided to request.

From is the sender address used by iris:reply/2 to send an asynchronous reply back.

State is the internal state of the iris_server.

Link is the relay connection to the Iris network if the handler needs additional communication.

If the function returns {reply, Reply, NewState}, the Reply will be given back to From as the return value to request. The iris_server then will continue executing using NewState.

If the function returns {noreply, NewState}, the iris_server will continue executing with NewState. Any reply to From must be sent back explicitly using iris:reply/2.

If the return value is {stop, Reason, Reply, NewState}, Reply will be sent back to From, and if {stop, Reason, NewState} is used, any reply must be sent back explicitly. In either case the iris_server will call Module:terminate/2 and terminate.

handle_publish/4 

       handle_publish(Topic, Event, State, Link) -> {noreply, NewState} | {stop, Reason, NewState}
           Topic    = [byte()]
           Event    = binary()
           State    = term()
           Link     = iris:connection()
           NewState = term()
           Reason   = term()

Whenever an iris_server receives an event sent using iris:publish/3, on a subscribed topic, this function is called to handle it.

Topic is the Topic argument provided to publish.

Event is the Event argument provided to publish.

State is the internal state of the iris_server.

Link is the relay connection to the Iris network if the handler needs additional communication.

If the function returns {noreply, NewState}, the iris_server will continue executing with NewState.

If the return value is {stop, Reason, NewState}, the iris_server will call Module:terminate/2 and terminate.

handle_tunnel/3 

       handle_tunnel(Tunnel, State, Link) -> {noreply, NewState} | {stop, Reason, NewState}
           Topic    = iris:tunnel()
           State    = term()
           Link     = iris:connection()
           NewState = term()
           Reason   = term()

Whenever an iris_server receives an inbound tunnel connection sent using iris:tunnel/3, this function is called to handle it.

Tunnel is the Iris data stream used for ordered messaging.

State is the internal state of the iris_server.

Link is the relay connection to the Iris network if the handler needs additional communication.

If the function returns {noreply, NewState}, the iris_server will continue executing with NewState.

If the return value is {stop, Reason, NewState}, the iris_server will call Module:terminate/2 and terminate.

Note, handling tunnel connections should be done in spawned children processes in order not to block the handler process. The handle_tunnel method itself is called synchronously in order to allow modifications to the internal state if need be.

handle_drop/2 

       handle_drop(Reason, State) -> {noreply, NewState} | {stop, NewReason, NewState}
           Reason    = term()
           State     = term()
           NewState  = term()
           NewReason = term()

Error handler called in the event of an unexpected remote closure of the relay connection to the local Iris node.

Reason is the encountered problem (most probably an EOF).

State is the internal state of the iris_server.

If the function returns {noreply, NewState}, the iris_server will continue executing with NewState, though no new data will arrive, nor can be sent.

If the return value is {stop, NewReason, NewState}, the process will call Module:terminate/2 and terminate.

terminate/2 

       terminate(Reason, State)
           Reason    = term()
           State     = term()

This method is called when an iris_server is about to terminate. It is the opposite of Module:init/1 and should do any necessary cleaning up. The return value is ignored, after which the handler process stops.

Reason is the term denoting the stop reason. If the iris_server is terminating due to a method returning {stop, ...}, Reason will have the value specified in that tuple.

State is the internal state of the iris_server.

Functions

start(Port, App, Module, Args) -> {ok, Server, Connection} | {error, Reason}

  • Port = pos_integer()
  • App = string()
  • Module = atom()
  • Args = term()
  • Server = pid()
  • Connection = connection()
  • Reason = atom()

Creates and starts an Iris server process.

The server will connect to the locally running Iris relay, and receive all the inbound network events, conforming to the iris_server behavior.

Port is the local TCP endpoint on which the Iris relay is listening.

App is the application group the server wishes to join.

Module is the name of the callback module

Args is an arbitrary term provided to Module:init/1

If the iris_server is successfully initialized, started and the relay node connection made, {ok, Server, Connection} is returned, where Server is the pid of the iris_server and Connection is the data link to the relay to allow communication from outside the iris_server handler too.

If Module:init/1 fails or the connection to a local Iris node cannot be made, the method returns with {error, Reason}.

If the iris_server is successfully initialized, started and the relay node connection made, {ok, Server, Connection} is returned, where Server is the pid of the iris_server and Connection is the data link to the relay to allow communication from outside the iris_server handler too.

stop(Server) -> ok | {error, Reason}

  • Server = pid()
  • Reason = atom()

Gracefully terminates the server process.

Server is the pid of the iris_server to terminate.
