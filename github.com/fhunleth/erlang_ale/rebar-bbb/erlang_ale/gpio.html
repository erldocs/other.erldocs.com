This is the implementation of the gpio interface module.

This is the implementation of the gpio interface module.

There is one process per GPIO pin. Each process opens a C-port that manipulates the hardware.

The one-process-per-pin design has been choosen since it allows to attach each pin to the appropriate place in the supervision tree of the entire application.

Types

interrupt_condition() = none | rising | falling | both

pin() = 0..63

pin_direction() = input | output

pin_state() = 0 | 1

Functions

start_link(X1::{pin(), pin_direction()}) -> {ok, pid()} | ignore | {error, term()}

starts a pin with the given direction.

To simplify things we aways identify the process for a pin by the pin number, so there is no need to remember the Pid of a pin process spawned.

release(Pin::pin()) -> ok

release/1 stops the pin process and frees resources on the hardware.

write(Pin::pin(), Value::pin_state()) -> ok | {error, writing_to_input_pin}

write/2 sets an output pin to the value given.

read(Pin::pin()) -> pin_state()

read/1 returns the value of a pin.

set_int(Pin::pin(), Condition::interrupt_condition()) -> ok | {error, Error}

  • Error = wrong_condition | term()

set_int/2 sets an interrupt on a pin with a given condition. The requesting process will be sent a message with the structure {gpio_interrupt, Pin, Condition} when the interrupt triggers. More than one process can listen on an interrupt condition, but only one interrupt condition can be set at a time.

from_port(Pin, Msg) -> term()

from_port/2 should not be called from the outside! It is used to transform the messages from a port into a message that is sent to the pin process using call instead of forcing us to handle the messages in handle_info/2.
