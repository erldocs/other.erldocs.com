ceck

Port/cnode mocking library for Erlang.

Port/cnode mocking library for Erlang. For applications that use a cnode or other port, it can be difficult to test, particularly in failure conditions. By using Erlang's Slave module as well as gen_server_mock, any number of responses and expectations can be created.

Ceck can be configured in two ways: either the usual OTP app files, of through set_remote/2 and set_remotes/1. These will start ceck as a proper application on the remote nodes. If a more quick and dirty set up is desired, ceck_node:start_link/1 can be used directly.

Types

mock_cnode_spec() = {node_name(), respondant_list()}

node_name() = atom()

respondant_list() = [atom()]

Functions

start() -> ok

Start ceck using an app or config file. If the current vm is not running in distributed mode, ceck starts it. Then, ceck proceeds to start the listed slave nodes, and start itself proper on those nodes. Note that ceck does not actually run on the base node itself.

start_testnode(Name::atom()) -> {ok, atom()} | {error, no_rsh | timeout}

Start a slave node using the default host.

See also: start_testnode/2.

start_testnode(Name::atom(), Host::string() | atom()) -> {ok, atom()} | {error, no_rsh | timeout}

Start a slave node using the given name and host. The slave node inherits the cookie and paths of the master node. If the erlang vm is not running in distributed mode, this will fail.

get_env(Key::any()) -> {ok, any()}

get_env/2 with a default of undefined. @see gen_env/2

get_env(Key::any(), Default::any()) -> {ok, any()}

Get the application env Key for ceck. If it's not found return Default.

set_env(Key::any(), Value::any()) -> ok

Sets the application env key Key for ceck to the given value.

set_remote(Node::atom(), Names::[atom()]) -> ok

Sets ceck to create a mock cnode with the given Node with the listed responders Names.

set_remotes(Proplist::[mock_cnode_spec()]) -> ok

A batch version of set_remote/2.
