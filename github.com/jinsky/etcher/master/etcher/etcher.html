compile(Source, Options) -> {ok, template()}

  • Source = chardata()
  • Options = [Option]
  • Option = {filters, [custom_filter()]} | {tags, [custom_tag()]} | {template_loaders, [template_loader()]}

Compiles unicode/UTF-8-encoded template source code into a template. You can do what you want with this term - store it in an mneisa database, call erlang:term_to_binary/1 on it and write it to a file.

Most template inclusion happens at runtime, but the 'include' tag will include templates at compile-time if the template path is specified as a string literal, like "{% include 'articles/123.txt' %}". To find templates at compile-time, you must provide a template_loaders option. It works exactly the same as the template_loaders option provided to render/3.

add_record_defs(RecDefs) -> ok

  • RecDefs = filename() | [{record_name(), [record_field()]}]

Use this function to inform the etcher application about records you would like to use in rendering contexts.

The easiest way to inform etcher about your record defintions is to pass in the name of a .hrl file containing those definitions.

You can also pass in a list of {record_name(), [record_fields()]} if it's more convenient.

Note that there is currently no way to get etcher to forget about record definitions it's already been informed about. This functionality will be added later.

render(Template, Context, Options) -> RenderedContent

  • Template = template() | template_path()
  • Context = context()
  • Options = [Option]
  • Option = {return, list} | {return, binary} | {auto_escape, bool()} | {url_mapper, function()} | {allowed_include_roots, PathList} | {template_loaders, [template_loader()]} | {compiler_opts, list()}
  • PathList = [filename()]
  • RenderedContent = chardata() | unicode_binary() | list()

Renders the Template using the Context you provide.

The returned content defaults to chardata(), but using the return option you can get a binary() ({return, binary}) or a list() ({return, list}).

If the Template argument is a template_path(), you will need to include a template_loaders option so that the template file (it's usually a file) can be located and loaded.

Auto-escape is on by default, as it is with Django. You can use use the {auto_escape, false} option to turn it off. You'll certainly want to do this if you are generating non-XML/non-HTML documents, such as emails.

The url_mapper option is specifically for use with the {% url ... %} tag. Django has it's own way of using this tag which ties in with the rest of it's framework. We obviously don't have a 'rest of a framework' so we'll have to do something a little different. If you pass in a fun() of Arity 1 as a url_mapper, then when the url tag is used, it's parameters will be passed to your fun as a list(). The parameter list will contain either one, or two parameters. If the tag looks like {% url required_param arg1,arg2,name1=value1 %}, then your UrlMapper fun() will receive ["required_param", "arg1,arg2,name1=value1"]. You are then responsible for returning a URL string. Note that no checking, or URL-encoding, will be done on the URL string you return. So, you need to make sure you get it right. There's a couple of url encoding functions exported from etcher_util that you can use if you need to.

allowed_include_roots is for the ssi standard tag (and any custom tag that might want to use it). It contains a list of absolute path prefixes for safe paths that ssi is allowed to use. It serves the same purpose as Django's ALLOWED_INCLUDE_ROOTS. By default allowed_include_roots is an empty list [].

Note, if you choose to use allowed_include_roots in custom tags, make sure to normalize the filepath before checking against these prefixes - prefix checking alone won't save you from paths like "/opt/safe/../../etc/passwd". You can use etcher_util:normalize_absolute_path/1 for this.

Use the template_loaders option to tell etcher how to find included/extended templates. Most template inclusion is handled at render-time rather than at compile time, and you will certainly need to provide this option if you are using template inheritence.

The compiler_opts option is used when templates are compiled on the fly during rendering. This happens more than you might think. For example, if you reference templates as a filename in an 'extends' tag, then the file will be compiled at render-time (unless it was pre-compiled and saved as a term_to_binary entity). The compiler_opts setting allows you to specifiy any of the options used in compile/2 and they will be used when compiling. One exception is the template_loaders option. You should specify this as an ordinary render option, and it will be used at compile time if it's needed.
