ebi_solver

This module is responsible for managing solver processes (ports).

This module is responsible for managing solver processes (ports). The module implements FSM that is acting as a connected process for the solver port.

Functions

start_link(Simulation) -> term()

Start the solver with the default port program.

See also: start_link/2.

start_link(Simulation, PortProgram) -> term()

Start the solver. The solver is started in the init state, waiting for a command to run the simulation.

run(FsmRef) -> term()

Run the actual simulation or resume it, if the simulation was suspended.

suspend(FsmRef) -> term()

Suspend the simulation, if it is running now.

cancel(FsmRef) -> term()

Cancel the simulation and delete all the related data. Use this in the case, when the simulation is not relevant anymore.

status(Pid) -> term()

Returns current status of the solver.

init(X1) -> term()

Initialize the FSM.

handle_event(X1, StateName, StateData) -> term()

The solver_cancel event can occur at any time during lifecycle of the solver.

handle_sync_event(X1, From, StateName, StateData) -> term()

Synchronous query for the current status can be invoked at any time.

handle_info(Info, StateName, StateData) -> term()

This function will be invoked for any other messages than normal events. The events will include: messages from the port. port exit status trap exit? Port exit statuses are forwarded to the corresponding FSM state functions. Normal exit messages ({'EXIT', ..}) from the port are ignored.

terminate(Reason, StateName, Fsm_state) -> term()

Termination of the FSM. Reasons can be the following: normal - when the solver is done with its job, shutdown - when the OTP application will be going down. _ - just in case. The final states can be the following: done - simulation was completed successfully. canceled - when the FSM was canceled by the user, failed - when the simulation fails (unstable scheme, etc),

code_change(OldVsn, StateName, StateData, Extra) -> term()

Handler for a code_change. There are no plans to use it.

init(X1, StateData) -> term()

Initialized solver. It can be either started or canceled.

running(X1, StateData) -> term()

Simulation is actually executed in this state. The solver can be suspended in this state.

restarting(X1, StateData) -> term()

The state restarting is considered for the cases, when someone is sending solver_run event while solver is in the suspending state.

suspending(X1, StateData) -> term()

The solver is requested to be suspended. Now we are waiting the solver to notify, that the simulation is stopped gracefully.

suspended(X1, StateData) -> term()

The solver is already suspended completely. One can resume it.
