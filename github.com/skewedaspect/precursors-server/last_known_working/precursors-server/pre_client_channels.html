pre_client_channels

Holds the channels for a client connection.

Holds the channels for a client connection. Backed by gen_event. Uses a simple callback model.

Each client gets its own pre_client_channels, but a channel can point to the same gen_server, or even simply have an internally handled state for each client. An hook is fired when a client connects, so that is a prime opportunity for a supervisor-style process to call set_channel/4, or set_sup_channel for a supervised handler.

The module passed into set_channel/4 or set_sup_channel must implement 3 functions: client_request/4, client_response/4, and client_event/3.

client_request/4 

  Module:client_request( Client :: pid(), Id :: any(), Request :: json(),
      Info :: any()) -> callback_return() ).
 
      Client :: process of the client connection.  Responses and events
          can be sent to this process directly.
      Id :: An opaque data type used for sending a response.
      Request :: The json request payload.
      Info :: The internal state of the callback channel.
          callback_return() can update this.

When the client makes a request to the channel, this callback is used. The callback can either have a response generated for it by sending a specific callback_return(), or it can send a response on it's own using the pre_client_connect:send/5 function.

client_response/4 

  Module:client_response( Client :: pid(), Id :: any(), Response :: json(),
      Info :: any() ) -> callback_return() ).
 
      Client :: process of the client connection.  New requests can be
          sent to the process directly.
      Id :: An opaque data type indication which request this response is
          for.  It is up to the callback module to keep track of the
          requests it has made.
      Response :: The json response payload.
      Info :: The internal state of the callback channel.
          callback_return() can update this.

The client has sent a response to a request the channel made (or at least it's claiming to).

client_event/3 

  Module:client_event( Client :: pid(), Event :: json(), Info() ) ->
      callback_return() ).
 
      Client :: Process of the client connection.  New requests or events
          can be sent to the process directly.
      Event :: The payload for the event.
      Info :: The internal state of the callbakc channel.
          callback_return() can update this.

The client has sent an event to the channel.

callback_return()

This can take many forms, many of which are the same as gen_event. They are documented here for great good.

'remove_handler'

Remove the particular channel callback from the client's channels.

{'ok', NewChannelState :: any()}

Updates the internal Info used by the callback functions.

{'ok', NewChannelState :: any(), 'hibernate'}

Same as above, but puts the channel handler into hibernation. This indicates it will not be processing new events for at least 2 garbage collection cycles.

{'swap_handler', Args1 :: any(), NewInfo :: any(), NewMod :: atom(), NewChannelState :: any()}

Swaps the callback module for a new one. Behaves the same as the gen_event return of the same signature.

{'reply', Payload :: json()}

Only valid when the callback function client_request/4 was used. Sends a response to the client using the same ID that came in over tcp.

{'reply', Socket :: 'tcp' | 'udp' | 'ssl', Payload :: json()}

Same as above, but allows one to specific which socket to use.
