YATSY: Yet Another Test Server - Yaws compatible

Yatsy is a simple test framework for testing your Erlang code. It makes it possible to execute test suites according to a defined structure and format, presenting the result either dynamically (via Yaws) or by generated HTML files.

A testcase is implemented by a function that you write. You collect a number of such testcases in an Erlang module which ends in: _SUITE.erl. You may have any number of such test suite modules which you compile to beam files located in your ebin directories.

You start your system, add the code paths to Yatsy and Yaws and start Yatsy. Yatsy will now automatically find all test suite modules that exist as beam files in you code path. Next you order Yatsy to run your test cases, suite by suite. The result will be accessible by pointing a Web browser to: http://localhost:8888/yatsy.yaws .

It is possible to control the behaviour of Yatsy. This can be done either via a list of {Key,Value} tuples, which is provided at startup, or via shell environment variables. The latter method makes it easy to write wrapper scripts that can start your system and Yatsy. You can configure:

If Yatsy should quit automatically when finished. If the test cases should run on a remote Erlang node. If static HTML files should be produced with the result.

There exist a number of predefined functions that will be called in every test suite module (M). When a new test suite starts, the function M:init_per_suite/1 will be called with the configuration lists (mentioned above). This function can return a list of new test suite specific configurations that each testcase in this particular test suite will get. Before each test case is run, the function M:init_per_testcase/2 will be called. It takes the test case name as the first argument and the configuration list as the second. Likewise, the corresponding functions M:fin_per_suite/1 and M:fin_per_testcase/2 will be called at the end.

The names of the test cases are defined by the list of atoms returned from M:all(suite). A list of documentation strings can be returned from M:all(doc). Each test case can also return a list of documentation strings via the test case function M:TestCase(doc).

Checklist, what you need to do:

Write Erlang code that tests the functionality of your system. Put the code in an xxx_SUITE.erl file. Make sure the xxx_SUITE.erl file is compiled. Checkout and compile Yatsy.
svn checkout http://yatsy.googlecode.com/svn/trunk/ yatsy
 Start your system and make sure that the code path includes the .../yatsy/ebin and ...yaws/ebin directories. Run: yatsy:quick(). Point a Web Browser to http://localhost:8888/yatsy.yaws and study the result from the test cases.

It is possible to add {Key,Value} tuples to the list of Config tuples when an init_XXXX routine returns. The following are some examples:

{timeout, Seconds} - This test case should use a non default timeout. {run_fin, Boolean} - Run the fin_XXXX routine (default: true)
